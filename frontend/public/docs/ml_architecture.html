    Bank Offer Feed Ranking — ML Architecture & Design

    A learning-oriented, code-grounded architecture reference for a production-aligned ML application that ranks personalized bank offers in a feed-style UI. This document maps every stage of the pipeline — from raw UCI Bank Marketing CSV ingestion to a FastAPI-served ranking response consumed by a React SPA — to the actual modules, files, and contracts in this repository.

    FastAPI · Pydantic v2 scikit-learn · HistGradientBoosting pandas · pyarrow · joblib React 19 · Vite · Tailwind v4 JSONL feedback store GitHub Actions CI

    A. Executive Overview

    What this system solves, and how every layer fits together.

    A.1 Problem & product framing

    The repository implements a personalized ranking system over a synthetic banking feed. Given a customer profile, it returns a top-K list of offers (term deposit reminder, savings boost, credit card upgrade, refinance prompt, advisor callback), each with a model score, normalized score, deterministic explanation, and full provenance (request, model, and feature versions). The product framing reframes a classic binary classification problem (UCI Bank Marketing bank-additional-full.csv) into a ranking + recommendation pattern, while preserving rigorous engineering practices: leakage controls, training/serving feature parity, contract-driven module boundaries, structured observability, and a JSONL feedback loop for future retraining.

    A.2 Seven-layer architecture at a glance

    The codebase organizes responsibility into seven explicit layers defined in specs/design.md, each owned by a distinct directory:

    # Layer Primary directories Responsibility
    1 Data ml/data/, shared/contracts/ Load, validate, normalize raw UCI CSV; emit cleaned Parquet artifacts.
    2 Synthetic Scenario ml/data/offers.py, pairs.py, labels.py Generate offer catalog, user-offer candidates, and deterministic labels.
    3 Feature ml/features/builder.py One FeatureBuilder shared between training and inference.
    4 Model ml/training/, ml/evaluation/ Train HistGradientBoostingClassifier, save artifacts, compute metrics.
    5 Ranking Application backend/app/services/ Candidate selection, scoring, reranking, explanation.
    6 Experience backend/app/api/, frontend/src/ FastAPI HTTP surface + React SPA feed and feedback UI.
    7 Observability & Delivery backend/app/core/logging.py, .github/workflows/, scripts/ Structured JSON logs, JSONL feedback store, three CI workflows, quality gate.

    B.3 Reproducible runbook

    The full offline pipeline is executed via three module entrypoints (as defined in docs/runbook.md and docs/install-local.md):

    # 1. Validate, preprocess, and extract sample users.
uv run python -m ml.data.run_pipeline

# 2. Generate offers, eligible pairs, and synthetic labels.
uv run python -m ml.data.run_synthetic

# 3. Train baseline ranker, evaluate, and persist artifacts.
uv run python -m ml.training.run_pipeline \
  --users  data/processed/bank_processed.parquet \
  --offers data/synthetic/offers.csv \
  --pairs  data/synthetic/user_offer_pairs.parquet \
  --schema shared/contracts/feature_schema.json
    Concept · Decision-time features

    The UCI duration field is recorded after the call ends, so it perfectly predicts conversion but is unavailable at decision time. The pipeline enforces this exclusion in three independent places — the data schema (shared/contracts/bank_data_schema.json with a leakage_note), the preprocess step (ml/data/preprocess.py), and the feature builder (ml/features/builder.py) — so a typo in one place still cannot leak the feature into training or serving.

    C. Data Engineering & Feature Layer

    How raw bytes become typed, leakage-safe, training-serving-consistent features.

    C.1 Ingestion & schema enforcement

    ml/data/loaders.load_raw_bank_data reads the semicolon-separated UCI CSV into a pandas DataFrame. ml/data/validation.validate_raw_data then loads the JSON Schema contract in shared/contracts/bank_data_schema.json and asserts:

    • All required columns (21 expected fields) are present.
    • The target column y contains no nulls.
    • Light dtype consistency checks (integer/number) emit warnings without crashing the pipeline (defensive ingestion).

    The schema contract is the canonical source of truth and embeds the leakage policy:

    {
  "default": {
    "target": "y",
    "excluded_fields": ["duration"],
    "leakage_note": "CRITICAL: The 'duration' field MUST be excluded by default from modeling. ...",
    "required_columns": [
      { "name": "age", "type": "integer" },
      { "name": "job", "type": "string" },
      { "name": "duration", "type": "integer" },
      { "name": "campaign", "type": "integer" },
      { "name": "euribor3m", "type": "number" },
      { "name": "y", "type": "string" }
    ]
  }
}

    C.2 Preprocessing & standardization

    ml/data/preprocess.preprocess_data performs four deterministic transformations:

    1. Excludes leakage fields (duration by default).
    2. Normalizes categoricals: .str.lower().str.strip() on every object column (UCI uses inconsistent casing and whitespace).
    3. Standardizes column names: replaces . with _ so emp.var.rate becomes emp_var_rate, which is also how the Pydantic schema SampleUserProfile models the field.
    4. Maps target: y becomes 1 / 0 from "yes" / "no".

    Cleaned data is persisted as Parquet for type fidelity and columnar reads, and a preprocessing summary is written to output/reports/preprocessing_summary.json for reproducibility (FR-005 in specs/requirements.md).

    C.3 Synthetic scenario generation

    The "rankable problem" is constructed on top of the cleaned dataset:

    • Offer catalog — ml/data/offers.py emits 5 typed offers (term deposit reminder, savings boost, credit card upgrade, refinance prompt, advisor callback) with structured eligibility_rules serialized as JSON strings into CSV.
    • Eligibility engine — ml/data/eligibility.check_eligibility is a pure function that takes a user dict + offer dict, evaluates rules (min_age, max_age, exclude_jobs, marital, education, housing, loan), and returns (is_eligible, reasons[]). It is reused identically at serve time by backend/app/services/candidate_service.py.
    • Pair generation — ml/data/pairs.generate_user_offer_pairs produces every eligible (user, offer) pair and reports coverage to output/reports/pair_generation_report.json.
    • Deterministic labels — ml/data/labels.generate_synthetic_labels applies offer-specific heuristics (e.g., credit card upgrade is positive for management/entrepreneur/admin/technician with no default) overlaid with hard negatives (campaign fatigue, credit default for credit offers) and a 5% hash-based deterministic flip for realism. The full assumption list is exported to output/reports/label_generation_report.json so reviewers know exactly what the model is being asked to learn.

    C.4 The shared FeatureBuilder (training-serving parity)

    The feature contract in shared/contracts/feature_schema.json declares four groups under "default": user_features (17), offer_features (4), user_offer_features (2), context_features (2), plus an explicit label and excluded_features: ["duration"].

    FeatureBuilder.build_features(df, is_training=...) is the single entrypoint used by training, evaluation, and the live ranking service. It enforces:

    • Deterministic column ordering (so the model sees the same schema layout offline and online).
    • Strict missing-column detection (raises ValueError on missing inputs).
    • Type coercion by group: numerical → pd.to_numeric(...).fillna(0.0), categorical → fillna("missing").astype(str), boolean → astype(bool).
    • Hard removal of any excluded_features that sneak in — a defense in depth for duration.
    • Label inclusion controlled by the is_training flag.
    Concept · Training-serving skew

    Skew happens when the offline feature pipeline differs (even subtly) from the online one — different null fills, different category encodings, or different column orders. Here, both paths import the same Python function with the same schema file, eliminating skew by construction rather than by discipline.

    C.5 No external feature store (intentional V1 scope)

    There is no Feast/Tecton/Redis feature store in V1 — this is explicit in specs/requirements.md §10 ("Out of Scope: Feature store infrastructure") and in ADR-0003. Instead, the system uses a shared feature module (the FeatureBuilder + a versioned JSON schema), which solves the consistency problem without the operational cost of a separate store. The preprocessor.json artifact written next to the model is the serving-time record of which schema generated the model.

    Key takeaway

    A schema file + a shared Python function is the lowest-cost form of a feature store. You get parity, versioning (via preprocessor.json), and contract checks without standing up infrastructure. Treat this as the "Level 1" you grow out of when real-time features and cross-team consumers arrive.

    D. Model Architecture & Training

    Why HistGradientBoostingClassifier, how it is trained, evaluated, and packaged.

    D.1 Model choice and rationale

    Per ADR-0001 (docs/adr/0001-model-choice.md), the baseline ranker is sklearn.ensemble.HistGradientBoostingClassifier, used in a point-wise formulation: each (user, offer) pair is a classification sample, and ranking is induced at inference by sorting predicted probabilities per user, then applying business reranking.

    Property Choice Why
    Family Histogram-based gradient-boosted trees Strong on tabular + mixed types, native missing & categorical support.
    Loss log_loss (binary classification default) Calibrated probabilities feed normalization & reranking.
    Hyperparameters max_iter=100, learning_rate=0.1, max_depth=6, random_state=42 Defined in ml/training/run_pipeline.py and echoed to the manifest.
    Categorical handling Native via categorical_features argument Avoids one-hot blowup; columns set to category dtype before fit.
    Serialization joblib.dump → baseline_model.joblib Stable scikit-learn artifact; loaded server-side without training imports.

    D.2 Training pipeline (ml/training/)

    ml/training/run_pipeline.main wires the full offline run:

    1. Load processed users (Parquet), synthetic offers (CSV), and labeled pairs (Parquet).
    2. Three-way merge on user_id ↔ id and offer_id.
    3. Normalize column aliases (label → y, eligible → is_eligible) and inject a default affinity_score.
    4. Group split via ml/training/split.split_data using sklearn.model_selection.GroupShuffleSplit with group_col="user_id" — a user appears strictly in train or val, never both.
    5. Call train_baseline_model, which builds features through the shared FeatureBuilder and fits the classifier.
    6. Persist artifacts and a run manifest, then evaluate on the validation split.

    Pointwise scoring & calibration

    For each candidate pair the model emits a probability:

    model_score = model.predict_proba(X)[:, 1]   # P(positive | features)

    At inference the RankingService normalizes these probabilities to [0, 1] within a single request: 

    normalized = (score - min(scores)) / (max(scores) - min(scores))   # within-request min-max

    This is a per-request rescaling, not a calibration across requests. It supports a stable visual sort and a downstream priority boost without altering relative order.

    Concept · Point-wise vs. pair/listwise ranking

    Point-wise treats each (user, offer) as an independent classification, then sorts at serve time. It is the simplest and cheapest formulation, and pairs cleanly with rules-based reranking. The repo intentionally leaves room for pair-wise or list-wise losses (e.g. NDCG-optimized) as a future iteration.

    D.3 Evaluation: classification + ranking

    ml/evaluation/evaluate.py reports both kinds of metrics. Classification metrics are global (roc_auc_score, log_loss); ranking metrics are computed per user, then averaged.

    For each user group (sorted by y_pred desc, take top-K):

    • Precision@K = (# positives in top-K) / K
    • Recall@K = (# positives in top-K) / total_positives_for_user
    • MAP@K — average of running precision at each positive hit in top-K, divided by min(K, total_positives)
    • NDCG@K = DCG@K / IDCG@K where DCG = Σ 1 / log2(rank + 2) on positive labels.

    Baseline numbers (committed in docs/model_card.md):

    Family Metric Value
    ClassificationAUC0.815
    ClassificationLog loss0.271
    Ranking @ K=3Precision@30.089
    Ranking @ K=3Recall@30.106
    Ranking @ K=3MAP@30.110
    Ranking @ K=3NDCG@30.110

    ml/training/train.save_artifacts writes three things side-by-side in ml/artifacts/: 

    {
  "run_id": "run_YYYYMMDD_HHMMSS",
  "timestamp": "2026-05-24T12:34:56",
  "model_type": "HistGradientBoostingClassifier",
  "hyperparameters": { "max_iter": 100, "learning_rate": 0.1, "max_depth": 6, "random_state": 42 },
  "feature_names": ["age", "job", ..., "is_eligible", "affinity_score"],

    "preprocessor": "ml/artifacts/preprocessor.json"
  },

    scripts/check_ml_artifacts.py validates this manifest in CI: required keys, artifact pointers, JSON validity of the preprocessor file, and existence (or git-ignored absence) of the binary model.

    Key takeaway

    Three artifacts always travel together — the binary model, the feature schema snapshot, and the run manifest. The serving layer needs all three to load successfully or it falls back. Treat them as one unit when copying to S3 or a registry.

    E.4 Service decomposition

    CandidateService

    Loads sample users + offer CSV, resolves the requested user, then iterates active offers calling the shared check_eligibility. For each eligible pair it computes a deterministic affinity_score (used as a feature and as a fallback signal) and explicitly drops duration from the payload as a final leakage guard.

    RankingService

    Builds features through FeatureBuilder(is_training=False), scores via predict_proba when artifacts are loaded, falls back to a heuristic otherwise (affinity + priority + recent-contact terms), normalizes scores within the request, and delegates to RerankService and ExplanationService.

    RerankService

    Externalized config (rerank_config.json): deduplicates by offer_id, applies priority_boost = priority_weight × 0.05 (toggleable), caps max_per_offer_type = 2 for diversity, and uses a stable tie-break by (-boosted_score, -normalized_score, offer_id).

    ExplanationService

    Returns a deterministic, template-based explanation per offer_type (5 templates + generic fallback) with a guaranteed safe_generate that never throws. ADR-0004 documents the explicit deferral of any LLM dependency in V1.

    E.5 Graceful degradation

    backend/app/core/model_loader.ModelArtifactLoader deliberately avoids importing the ML training package and instead loads the joblib model, the preprocessor.json, and run_manifest.json with per-file ArtifactStatus. If any are missing or corrupt:

    • /health still returns 200 with model_ready: false.
    • RankingService._fallback_score serves a deterministic heuristic so the UI continues to render a ranked feed.
    • model_version in the response is "fallback-v1" for traceability.
    • A warnings array is populated in the response.

    E.6 Feedback & impressions (the retraining seed)

    On every successful rank, FeedbackService.persist_impressions appends one line per displayed offer to data/processed/feedback_events.jsonl with event_type="impression", rank_position, and full provenance. User actions (viewed / clicked / accepted / dismissed / not_interested) are appended as event_type="feedback". ADR-0002 captures the rationale for JSONL: append safety, zero infra, and Pandas/Polars readiness for V2 retraining.

    Caveat · Latency & concurrency

    V1 is single-process, synchronous, and writes to a single JSONL file with an "a"-mode append. This is fine for a demo and CI but is not safe under high concurrency on a shared filesystem. ADR-0002 flags this as accepted V1 debt — graduate to a queue + warehouse (e.g. Kinesis + S3/Delta) before scaling.

    const API_BASE_URL =
  import.meta.env.VITE_API_BASE_URL || "http://localhost:8000";

async rankOffers(userId: string, debug = true): Promise<RankResponse> {

    G. Compute, Environment & Infrastructure

    Runtime, dependencies, deployment patterns, and observability.

    G.1 Runtime characteristics

    • CPU only. No GPU — gradient-boosted trees on tabular data.
    • Memory. ~100K row training and inference fit comfortably in a few hundred MB.
    • Concurrency. Single-process FastAPI (uvicorn); services are instantiated module-side and reused across requests.
    • Storage. Local filesystem for artifacts and the JSONL feedback store; optional S3 for artifact transfer in AWS.

    G.4 Deployment patterns

    V1 ships without Docker by design (ADR-0003 + docs/install-aws.md). Two paths are supported:

    Local development

    uv sync --extra dev for Python, npm install for the frontend, three pipeline modules to generate artifacts, then uvicorn app.main:app --reload --port 8000 and npm run dev. End-to-end driven by docs/install-local.md and docs/runbook.md.

    AWS single EC2

    t3.medium+ Amazon Linux/Ubuntu, ports 22/8000/5173, optional S3 bucket for shipping ml/artifacts/ via an instance IAM role (s3:GetObject). Static frontend is built with VITE_API_BASE_URL=http://<ec2>:8000 npm run build and served via npx serve or Nginx. No ALB, no RDS, no ECS — explicitly demo scope.

    infra/ is a scaffold for future Terraform — there is no committed IaC stack yet; the AWS guide reflects what is currently supported.

    G.6 Observability

    backend/app/core/logging.StructuredJsonFormatter emits one JSON line per event with a fixed envelope: 

    {
  "timestamp": "2026-05-25T17:09:11.123Z",
  "level": "INFO",
  "logger": "app.api.rank",
  "message": "rank response returned",
  "environment": "development",
  "request_id": "f3c2…",
  "correlation_id": "…",
  "model_version": "run_20260524_123456",
  "feature_version": "feature-default-v1",
  "event_type": "rank_response_returned",
  "results_count": 5
}

    HTTP middleware in backend/app/main.py binds x-request-id and x-correlation-id headers to context variables (ContextVar), so every downstream log line carries the same identifiers without explicit passing. log_event(...) guarantees that every emission includes event_type, model_version, and feature_version (defaulting to "unknown") — the three fields that make a log line useful for ML triage.

    Product metrics are computed lazily from the JSONL store by FeedbackService.generate_metrics_report and surfaced at GET /api/v1/feedback/metrics: impressions, feedback events, accept proxies (accepted | clicked), and dismiss proxies (dismissed | not_interested).

    Key takeaway

    One log envelope. Three required ML fields. request_id, model_version, feature_version let you tie any production incident back to a specific run manifest and a specific feature schema. Engineer this in early — backfilling it later is painful.

    H. Cross-Cutting Concerns & Design Rationale

    The decisions that shaped this codebase — and the debt they create.

    H.1 Architecture Decision Records (ADRs)

    ADRDecisionTrade-off accepted
    0001-model-choice.md HistGradientBoostingClassifier baseline Pointwise only; ranking is sort + rerank rather than NDCG-optimized.
    0002-feedback-storage.md Local JSONL feedback store Single-node concurrency; no SQL aggregations.
    0003-react-spa.md React + Vite SPA, no SSR No SEO/first-paint benefits; depends on CORS + backend availability.
    0004-llm-deferral.md Deterministic templated explanations Stylistically robotic explanations; LLM upgrade deferred to V2.

    H.2 Known limitations & technical debt

    • Synthetic labels. Heuristic + 5% deterministic noise (md5(user_id+offer_id) % 100) — strong inductive bias toward the rule set.
    • Static model. No online learning; the model is recompiled offline. Feedback is captured but not yet wired into a retraining job.
    • Single-process serving. No queue/worker pattern; rank requests are blocking and write to a shared JSONL file.
    • No batch inference. V1 intentionally ships only real-time scoring.
    • No experiment tracking platform. Runs are captured by run_manifest.json; there is no MLflow/W&B integration yet (deliberate per specs/requirements.md §10).
    • No infra-as-code. infra/ is a placeholder; AWS is documented but not Terraformed.
    • Two response shapes for rerank_factors. The Pydantic model is permissive (dict[str, Any]); long-term, lock it to a typed shape.

    H.3 Future improvements (roadmap-worthy)

    • Pair/listwise objective. Swap the classifier for LightGBM LambdaRank using the same FeatureBuilder.
    • Hybrid candidate generation. Add a retrieval stage (e.g. embedding-based) before eligibility filtering.
    • Online feature store. Replace ad-hoc affinity computations with a managed online feature service when latency matters.
    • Feedback-driven retraining. Schedule a job that reads feedback_events.jsonl, derives labels, retrains, and publishes a new run_manifest.json.
    • LLM explanation rewrite. Pass the deterministic template + top features to a small LLM for stylistic phrasing; keep the deterministic version as a safety fallback.
    • Containerization. Multi-stage Docker for backend + a static frontend image behind Nginx; deploy to ECS/Fargate.
    • Observability stack. Ship the JSON logs to a tracing/aggregation backend (LangSmith / OpenTelemetry / CloudWatch).

    H.4 Concepts worth remembering

    Train/serve parity by shared function

    A schema + a single Python function reused in both pipelines is the cheapest, most reliable way to prevent skew without a feature store.

    Defense in depth for leakage

    Encode the same constraint (here, drop duration) in the schema, the preprocess step, the feature builder, and the candidate service. Redundant checks are cheap and have saved countless models.

    Group-aware splits

    If a user has multiple rows, splitting randomly leaks information across train/val. Use GroupShuffleSplit(group=user_id) as this repo does.

    Per-request normalization ≠ calibration

    Min-max within a request stabilizes UI sort and lets reranking blend. It does not calibrate scores across requests — don't compare them across users.

    Externalized rerank config

    Business rules belong in JSON, not code. Diversity caps, priority weights, and fairness constraints should change without retraining the model.

    Graceful degradation by default

    Always carry a deterministic fallback path. A missing artifact should produce a degraded response, never a 500.

    Bottom line

    This repository is intentionally small, fully reproducible, and built around contracts. If you remember three things: (1) the same FeatureBuilder runs offline and online; (2) every prediction is traceable via request_id + model_version + feature_version in the structured logs and the response; and (3) the JSONL feedback store is the seed for V2 retraining — you can navigate the rest from the code.
